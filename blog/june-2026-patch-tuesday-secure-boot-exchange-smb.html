June 2026 Patch Tuesday: What an SMB Should Patch This Week (and the Secure Boot Deadline to Watch)

Ulises Paiz, Founder, Ghosxt
TL;DR — a quiet Patch Tuesday with one real deadline

June's Patch Tuesday is a rare zero-day-free month, which is good news but not a reason to skip it. Two things still need attention: the permanent fix for the Exchange Outlook Web Access flaw that has been running on emergency mitigation, and a hard Secure Boot certificate deadline at the end of the month that needs to be confirmed on every device.

  • No zero-days this month. Nothing in the release was being actively exploited when it shipped — unusual and welcome, but the clock starts the moment a patch is public.
  • Exchange OWA gets its permanent fix. CVE-2026-42897, the Outlook Web Access flaw that had been held off with Microsoft's Emergency Mitigation Service, is properly patched this month. On-prem Exchange shops: apply it.
  • SharePoint RCE rolled in. CVE-2026-45659, a remote-code-execution fix first shipped out-of-band on May 21, is folded into this release.
  • The Secure Boot deadline is the real story. The 2011-era certificates underpinning Secure Boot reach end of validity this cycle, at the end of June. Updated certificates need to be confirmed on every machine before then.

What a small business should do this week:

  1. Let Windows Update install this month's updates and reboot — on every PC and every server, not just the easy ones.
  2. If you run on-prem Exchange or SharePoint, prioritize those server updates now.
  3. Confirm the Secure Boot certificate update has actually applied to each device before the end-of-June cutoff.

This continues the May 2026 Patch Tuesday post; the managed IT page covers how we run patching so it is verified, not just hoped.

Microsoft shipped its June 2026 security updates today, the second Tuesday of the month, on schedule as always. Most months I lead this write-up with the one or two flaws that are already being exploited, because those are the ones that turn a slow patch cycle into a breach. This month is different, and in a good way: June is one of the rare Patch Tuesdays with no zero-days at all. Nothing in the release was under active attack or publicly disclosed before the fix arrived.

That does not mean there is nothing to do. It means the work this month is about staying current and hitting one specific deadline rather than racing an active exploit. There are still two items a Central Coast small business should care about: a long-running Exchange flaw finally getting its permanent fix, and a Secure Boot certificate deadline at the end of June that is easy to miss precisely because it is not a flashy CVE. Here is the plain-language version.

"No zero-days" is good news, with an asterisk

A zero-day is a vulnerability that attackers are already using, or that has been made public, before a patch exists, so defenders are starting from behind. A month with none means Microsoft got ahead of the attackers on everything in this release. That is genuinely reassuring and, frankly, a nice break from the recent run of months where something was already being exploited on day one.

The asterisk is important, though. The day a patch ships, the vulnerability it fixes stops being secret. Attackers download the same update everyone else does, compare it against the old code to find exactly what changed, and build exploits aimed at the organizations that have not yet applied it. This is why so many real breaches use vulnerabilities that had a fix available weeks or months earlier: the danger is rarely the zero-day, it is the patch gap. A quiet month is the easiest time to close that gap, not a reason to leave it open. The point of patching on a schedule, covered in the 10 essentials, is that it works only when it is consistent.

The Exchange Outlook Web Access flaw finally gets a real patch

If you have been following along, CVE-2026-42897 is not new. It is the cross-site-scripting flaw in Outlook Web Access on on-premises Exchange Server that we wrote about when it first surfaced, in the Exchange zero-day post. Rated Critical, it had been held at bay by Microsoft's Exchange Emergency Mitigation Service, the automatic, on-by-default stopgap that buys time but is not a cure. This month it gets its permanent fix.

For the small businesses still running their own Exchange server, this is your priority update. Emergency mitigations are exactly that, temporary, and they can be disabled, fail to apply, or be incomplete; the actual patch is what closes the door. If you are not sure whether you still run on-prem Exchange or whether the mitigation has been active on your server, that uncertainty is itself worth a phone call. It is also one more nudge toward the cloud-hosted email that removes this whole category of server-maintenance risk, the trade-off we walk through in the cloud migration guide.

A SharePoint server fix, folded in

The other server-side item is CVE-2026-45659, a remote-code-execution vulnerability in Windows SharePoint Server rated 8.8, which Microsoft first addressed in an out-of-band update on May 21 and has now rolled into the June Patch Tuesday release. SharePoint RCE flaws are a recurring theme, as we noted with a different one in the May SharePoint post, and the pattern is consistent: if you run SharePoint on your own server, these are high-priority because a remote-code-execution flaw lets an attacker run their code on the machine. Most small businesses that use SharePoint do so through Microsoft 365, where Microsoft patches the back end for you; this one is aimed at the shrinking number who still self-host.

The real deadline this month: Secure Boot certificates

Here is the item with an actual date attached, and the one most likely to be overlooked because it is not a vulnerability in the usual sense. Secure Boot is the mechanism that stops malicious code from loading before Windows even starts, a foundational protection against some of the nastiest, hardest-to-detect malware. It works by trusting boot components that are cryptographically signed with certificates, and the certificates most devices have relied on were issued back in 2011.

Those 2011-era certificates are now reaching the end of their validity through 2026, and this cycle lands at the end of June. Microsoft has been distributing updated certificates so that devices keep an unbroken chain of trust, but, and this is the catch, the rollout is not silently guaranteed on every machine. A device that does not receive the updated Secure Boot certificates in time can lose the ability to trust newly signed boot components and to take certain boot-related security updates, quietly eroding a protection most owners do not even know they have.

For a small business, this is not a reason to panic, but it is a reason to act deliberately. The right move is to make sure this month's updates are installed and then confirm, device by device, that the Secure Boot certificate update has actually applied before the end-of-June cutoff. That verification step is the part that gets skipped, because "Windows Update ran" is not the same as "the certificate update landed and Secure Boot is healthy." On a handful of machines this is manageable by hand; across a couple dozen devices and a server or two, it is exactly the kind of thing a managed IT setup tracks and proves rather than assumes.

What a small business should actually do this week

  • Install and reboot, everywhere. Let Windows Update apply this month's updates on every PC and every server, and reboot when prompted. The machine someone keeps deferring is the one that bites you.
  • Prioritize on-prem Exchange and SharePoint. If you self-host either, apply the CVE-2026-42897 and CVE-2026-45659 fixes now rather than at the end of your normal cycle.
  • Verify Secure Boot certificates. Confirm the certificate update has applied on each device before the end-of-June deadline, not just that updates "ran."
  • Don't forget the orphans. The old PC running one critical app, the server in the closet, the laptop that lives in someone's bag, those are where patch gaps hide.
  • Keep your backups current. Patching reduces the odds of an incident; a tested backup is what saves you when one happens anyway. See the backup and DR post.

FAQs about the June 2026 Patch Tuesday

Were there any zero-days in the June 2026 Patch Tuesday?

No, and that makes June unusual. This month's release shipped with no zero-day vulnerabilities, meaning nothing in it was being actively exploited or had been publicly disclosed before the patch arrived. That is a genuinely quiet Patch Tuesday and good news, but it is not a reason to skip the updates. A zero-day-free month simply means attackers were not yet using these specific flaws when Microsoft published the fix; once the patch is out, the vulnerabilities are public knowledge and criminals begin reverse-engineering them to attack the businesses that delay. The window between a patch being released and being applied is exactly when "not exploited" turns into "exploited." Quiet months are the easy ones to get current on; take advantage of it.

What's the most important thing to patch this month?

For most small businesses, the standard Windows and Office updates through Windows Update cover the essentials, and this month that includes the permanent fix for the Exchange Outlook Web Access flaw (CVE-2026-42897) that had been held off with emergency mitigation, plus a SharePoint Server remote-code-execution fix (CVE-2026-45659) rolled into the release. If you run an on-premises Exchange or SharePoint server, those are your priority and should be applied promptly. But the item with an actual deadline this month is not a CVE at all: it is the Secure Boot certificate update, which has a hard cutoff at the end of June and needs to be validated on every device before then.

What is the Secure Boot certificate expiration, and do we need to worry about it?

Secure Boot is the protection that stops malicious code from loading before Windows starts, and it works by trusting components signed with certificates that were issued back in 2011. Those original certificates are now reaching the end of their validity in 2026, with this cycle landing at the end of June. Microsoft has been distributing updated certificates so devices keep a valid chain of trust, but the rollout is not automatic and guaranteed on every machine, which is the catch. A device that does not receive the updated certificates in time can lose the ability to trust newly signed boot components and to receive certain boot-related security updates, quietly weakening one of its foundational protections. For a small business this is not a fire drill, but it is a real deadline that needs someone to confirm, device by device, that the certificate update has actually applied.

We're a small business with no IT staff. How do we keep up with Patch Tuesday every month?

The honest answer is that doing it well by hand is hard, which is why most small businesses either fall behind or never confirm that updates actually landed. The practical baseline is to make sure Windows Update is enabled and not endlessly deferred on every machine, reboot when asked rather than clicking "remind me later" for weeks, and not forget the servers and the one old PC running a critical app that everyone avoids touching. Beyond that baseline, managed patch management is exactly the kind of unglamorous, high-value work a managed IT provider handles: updates are deployed centrally, tested, and then verified across every device, including the servers and the edge cases, so "we patch monthly" becomes something you can actually prove rather than hope.

If there are no zero-days this month, can we skip the updates?

No. The absence of zero-days describes the state of the world on the day the patch shipped, not the day after. Once Microsoft publishes the fixes, the underlying vulnerabilities are documented and attackers actively study them to build exploits aimed at the businesses that did not patch, which is why so many real-world breaches use vulnerabilities that had a fix available months earlier. Skipping a quiet month also lets you drift out of a routine that only works when it is consistent. And this particular month carries the Secure Boot certificate deadline, so skipping it has a concrete downside beyond the usual. Treat every Patch Tuesday as routine maintenance; the quiet ones are simply easier to stay on top of.
